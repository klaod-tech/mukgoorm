# Step 3 — 음식_상세: 메뉴 선택 로짓 업데이트

> ⬅️ [목차](n8n_ml_nodes.md) | ⬅️ [Step 2 완료 후](n8n_02_피드백_로짓.md)

> **대상 워크플로우:** `음식_상세` (또는 현재 `/webhook/food/select`를 담당하는 워크플로우)

---

## 목표

유저가 식당 메뉴를 **실제로 선택**할 때 해당 카테고리의 로짓을 **+0.1** 업데이트합니다.

붐업/붐다운(±0.2)보다 약한 신호 — 명시적 피드백보다는  
행동 기반의 약한 선호 신호이기 때문입니다.

### 수정 전 흐름
```
메뉴 선택 입력(Webhook) → 방문 기록 저장(Supabase) → 선택 응답 포맷(Code) → 선택 완료 출력(Respond)
```

### 수정 후 흐름
```
메뉴 선택 입력(Webhook)
    → 방문 기록 저장(Supabase)         ← 기존 유지
    → 선택 응답 포맷(Code)             ← 기존 유지
    → [메뉴선택 현재 로짓](Supabase)   ← 신규 추가
    → [메뉴선택 로짓 계산](Code)       ← 신규 추가
    → [메뉴선택 로짓 저장](HTTP)       ← 신규 추가
    → 선택 완료 출력(Respond)
```

## 3-3. 메뉴선택 현재 로짓 노드 추가

`선택 응답 포맷` 오른쪽 **+** → `Supabase` 추가

| 항목 | 값 |
|------|-----|
| **Operation** | Get Many |
| **Table** | `user_preference_logits` |
| **Return All** | ✅ 체크 |

**Filters:**
```
Key Name  : user_id
Condition : equal
Key Value : {{ ($('메뉴 선택 입력').item.json.body || $('메뉴 선택 입력').item.json).user_id }}
```

노드 이름: **`메뉴선택 현재 로짓`**

## 3-4. 메뉴선택 로짓 계산 노드 추가

`메뉴선택 현재 로짓` 오른쪽 **+** → `Code` 추가

```javascript
// ────────────────────────────────────────────────
// 메뉴선택 로짓 계산 노드
// ────────────────────────────────────────────────
// 목적:
//   유저가 메뉴를 실제로 선택했을 때 해당 카테고리 로짓 +0.1
//
// 왜 0.1인가:
//   - 월드컵 승리: +0.5 (강한 명시적 선호)
//   - 붐업:       +0.2 (명시적 선호)
//   - 메뉴 선택:  +0.1 (행동 기반 약한 신호)
//   행동은 의도보다 약하기 때문에 학습률을 낮게 설정
//
// category가 없으면 로짓 업데이트를 건너뜁니다.
// ────────────────────────────────────────────────

const raw    = $('메뉴 선택 입력').first().json
const body   = raw.body || raw

const userId   = body.user_id
const category = body.category   // 없으면 undefined

// category 없으면 로짓 업데이트 불필요 → 빈 배열 반환
if (!category) {
  return [{ json: { skipped: true, reason: 'category 없음' } }]
}

const ALPHA = 0.1   // 메뉴 선택 학습률

// 현재 카테고리 로짓 값 찾기
const allLogits = $input.all()
const existing  = allLogits.find(i => i.json.category === category)

const currentLogit = existing ? (existing.json.logit ?? 0) : 0
const currentCount = existing ? (existing.json.sample_count ?? 0) : 0

return [{
  json: {
    user_id:      userId,
    category:     category,
    logit:        Math.round((currentLogit + ALPHA) * 1000) / 1000,
    sample_count: currentCount + 1,
    updated_at:   new Date().toISOString()
  }
}]
```

노드 이름: **`메뉴선택 로짓 계산`**

## 3-5. 메뉴선택 로짓 저장 노드 추가 (HTTP Request)

`메뉴선택 로짓 계산` 오른쪽 **+** → `HTTP Request` 추가

| 항목 | 값 |
|------|-----|
| **Method** | POST |
| **URL** | `https://[프로젝트ID].supabase.co/rest/v1/user_preference_logits` |
| **Authentication** | Predefined Credential Type |
| **Credential Type** | Supabase API |
| **Credential** | Supabase account 선택 |

**Headers:**

| Name | Value |
|------|-------|
| `Prefer` | `resolution=merge-duplicates` |
| `Content-Type` | `application/json` |

**Body:**
- **Body Content Type**: Raw
- **Content Type**: `application/json`
- **Body**:

```
={{ JSON.stringify([$json]) }}
```

> ⚠️ `skipped: true`가 반환된 경우에도 HTTP 요청이 실행됩니다.  
> Supabase는 body의 데이터가 불완전하면 에러를 내지만  
> 실제로는 category가 있어야 웹훅을 보내므로 건너뜀 케이스는  
> 운영 중에는 거의 발생하지 않습니다.

노드 이름: **`메뉴선택 로짓 저장`**

## 3-7. 전체 흐름 확인

```
메뉴 선택 입력(Webhook)
    ↓
방문 기록 저장(Supabase - Create, restaurant_log)    ← 기존
    ↓
선택 응답 포맷(Code)                                  ← 기존
    ↓
메뉴선택 현재 로짓(Supabase - Get Many)               ← 신규
    ↓
메뉴선택 로짓 계산(Code - +0.1)                       ← 신규
    ↓
메뉴선택 로짓 저장(HTTP Request - Upsert)             ← 신규
    ↓
선택 완료 출력(Respond to Webhook)
```

**Save → Active 확인**
