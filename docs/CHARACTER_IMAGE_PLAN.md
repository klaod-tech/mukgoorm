# 캐릭터 이미지 개인화 시스템 기획

> 작성일: 2026-05-14
> 상태: 기획 완료 / 발표용 시연은 사전 제작 이미지로 대체
> 실제 구현 전제: GPT 이미지 생성 API 접근 권한 필요 (현재 학교 공용 키로 불가)

---

## 1. 전체 개요

유저가 앱을 처음 시작할 때 **음식 이상형 월드컵**을 통해 선호도를 수집하고,
그 결과를 바탕으로 **AI가 개인화된 미소녀 캐릭터 이미지를 생성**한다.

생성에 시간이 걸리는 문제는 "큐브 → 캐릭터 진화" 스토리텔링으로 해결한다.

```
온보딩 시작
  → 음식 이상형 월드컵 (16강, 4라운드)
  → 선호도 데이터 저장 + 우승 음식 확정
  → 사용자 이미지 업로드 (선택)
  → AI 이미지 생성 큐 등록 (백그라운드)
  → 앱 사용 시작 (큐브 상태)
  → 1일 후 / 생성 완료 시 → 큐브 → 캐릭터 진화 애니메이션
```

---

## 2. 음식 이상형 월드컵

### 방식
- **16강 토너먼트** (4라운드, 총 15번의 선택)
- 매 라운드 음식 이미지 2장을 나란히 보여주고 클릭으로 선택
- 진행률 바 표시 (예: "4/15")

### 수집 데이터
매 선택마다 아래를 저장:

| 필드 | 내용 |
|------|------|
| `winner` | 선택된 음식 |
| `loser` | 탈락한 음식 |
| `round` | 라운드 번호 (1~4) |
| `category_winner` | 선택된 음식 카테고리 (한식/중식/양식/분식/디저트 등) |
| `category_loser` | 탈락한 음식 카테고리 |

### 후보 음식 풀 (예시, 16개)
```
한식: 삼겹살, 김치찌개, 비빔밥, 순대국밥
중식: 짜장면, 짬뽕, 탕수육, 마라탕
양식: 피자, 스테이크, 파스타, 햄버거
분식/기타: 치킨, 족발, 떡볶이, 아이스크림
```

### 선호도 분석 결과 저장 (worldcup_result 테이블)
```sql
CREATE TABLE worldcup_result (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         TEXT NOT NULL,
  champion        TEXT NOT NULL,        -- 우승 음식
  top4            TEXT[],               -- 4강 진출 음식들
  preferred_categories TEXT[],          -- 많이 선택한 카테고리 순
  aversion_categories  TEXT[],          -- 많이 탈락시킨 카테고리
  selections      JSONB,                -- 전체 선택 기록 배열
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

## 3. AI 이미지 생성 파이프라인

### 입력 데이터 (프롬프트 구성 재료)
1. 월드컵 우승 음식 + 선호 카테고리
2. (선택) 사용자가 업로드한 참고 이미지

### 이미지 생성 목표
- **종류별 15~20장** 생성 (표정/포즈/배경 등 변형)
- 모델: GPT-4o 이미지 생성 (gpt-image-1 또는 dall-e-3)
- 장당 예상 소요: 최대 5분
- 총 소요: 단독 사용 시 75~100분 / 동시 접속자 多 시 최대 2시간

### 프롬프트 예시
```
우승: 치킨, 선호: 한식·분식 계열
→ "귀여운 미소녀 캐릭터, 치킨 테마, 따뜻하고 편안한 색감, 
   한국 음식 배경 소품, 애니메이션 스타일, 고퀄리티"
```

### 생성 큐 관리 (image_generation_queue 테이블)
```sql
CREATE TABLE image_generation_queue (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     TEXT NOT NULL,
  status      TEXT DEFAULT 'pending',  -- pending / processing / done / failed
  prompt      TEXT NOT NULL,
  image_urls  TEXT[],                  -- 완성된 이미지 URL 배열
  enqueued_at TIMESTAMPTZ DEFAULT now(),
  completed_at TIMESTAMPTZ
);
```

## 4. 큐브 → 캐릭터 진화 시스템 (시간 벌기 UX)

### 개념
이미지 생성 대기 시간을 "캐릭터가 알에서 부화하는 과정"으로 포장.

```
[Day 0] 월드컵 완료 → 큐브(알) 형태로 앱 시작
[생성 중] 큐브에 균열 애니메이션 표시 / 진행도 힌트
[완료] 큐브 깨지는 연출 → AI 생성 캐릭터 첫 등장
```

### 상태 단계
| 단계 | 조건 | 표시 이미지 |
|------|------|------------|
| 큐브 | 이미지 생성 미완료 | `/cube.png` (기본 회색 큐브) |
| 균열 | 50% 이상 완료 | `/cube_crack.png` |
| 부화 중 | 90% 이상 완료 | `/cube_hatching.gif` |
| 진화 완료 | 생성 완료 | AI 생성 이미지 중 대표 1장 |

### 트리거 조건
- 이미지 생성 완료 (`image_generation_queue.status = 'done'`)
- OR 최초 사용 후 24시간 경과 (어느 쪽이든 먼저 도래한 시점)

## 5. 발표 시연 계획 (실제 API 미사용 대체안)

> **이유**: 현재 사용 중인 GPT API 키가 학교 공용 키여서 이미지 생성 API 인증 불가.

### 대체 방식
1. **사전 제작 이미지 세트** 준비 (캐릭터별 15~20장)
   - `public/demo_images/` 폴더에 저장
   - 실제 완성 이미지처럼 보이도록 캐릭터 테마별로 분류

2. **발표 흐름**:
   ```
   월드컵 시연 (실제 동작)
     → "이제 AI가 이미지를 생성합니다" 설명
     → 큐브 → 진화 애니메이션 시연 (사전 제작 이미지로 트리거)
     → "실제 서비스에서는 GPT 이미지 API로 개인화 이미지 생성"
   ```

3. **구현 완성 기준**:
   - 월드컵 UI 및 선택 데이터 저장 ✅ 구현 목표
   - 큐브 → 진화 진행 상태 표시 ✅ 구현 목표
   - AI 이미지 생성 실제 호출 → 발표 후 API 확보 시 추가

## 6. 구현 우선순위

| 항목 | 우선순위 | 비고 |
|------|---------|------|
| 월드컵 UI + 선택 기록 저장 | ★★★ 필수 | 발표 핵심 시연 |
| 큐브/진화 이미지 상태 전환 | ★★★ 필수 | 이미 Home.tsx에 캐릭터 이미지 로직 존재 |
| 선호도 분석 결과 저장 | ★★☆ 중요 | worldcup_result 테이블 필요 |
| 사전 제작 이미지 세트 | ★★☆ 중요 | 발표 시연용 |
| AI 이미지 생성 실제 호출 | ★☆☆ 추후 | API 키 확보 후 진행 |
| 이미지 생성 큐 관리 | ★☆☆ 추후 | 백엔드 작업 필요 |
