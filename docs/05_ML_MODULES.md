## GPT vs ML 역할 분담 원칙

```
먹구름봇 (오케스트레이터)  →  GPT 영구 유지
  - 자연어 파싱 ("어제 저녁에 치킨 먹었어" → 의도 추출)
  - 봇 트리거 의도 분류 ("광교산에서 샌드위치" → 식사봇 + 일기봇)
  - 대사/코멘트 생성 (날씨, 배지, 주간 리포트)
  → 자연어 이해는 대체 불가 — GPT 계속 사용

전문봇들 (식사봇 / 메일봇 / 체중관리봇 ...)  →  ML 점진적 대체
  - 반복적·구조화된 작업만 담당
  - GPT가 생성한 레이블을 DB에 누적 → ML 학습 → GPT 대체
  → 비용 절감 + 개인화 강화
```

## pattern.py — 식습관 패턴 분석

**역할**: DB meals에서 최근 14일 데이터 분석 → 패턴 탐지 → GPT System Prompt에 자연어로 주입

**탐지하는 5가지 패턴**:
1. **요일별 과식**: 특정 요일에 목표 110% 초과가 50% 이상 반복
2. **아침 결식**: 14일 중 7일 이상 아침 기록 없음
3. **저녁 집중 섭취**: 저녁 칼로리가 하루 평균 50% 이상
4. **주간 추이**: 이번 주 평균 vs 지난 주 평균 100kcal 이상 차이
5. **연속 소식**: 3일 연속으로 목표의 67% 미만 섭취

**사용 위치**: `utils/embed.py` → 📋 하루 정리 버튼의 GPT 대사 생성 시

## ml.py — 칼로리 보정 모델

**역할**: GPT-4o 추정 칼로리를 양 표현 패턴 + 개인화 모델로 보정

**즉시 보정 (모델 없이 동작)**:

| 표현 | 배율 |
|------|------|
| 조금/약간/살짝/적게 | ×0.7 |
| 반/절반 | ×0.5 |
| 한 그릇/보통 | ×1.0 |
| 많이/가득 | ×1.4 |
| 두 그릇 | ×2.0 |
| 엄청/대식 | ×1.8 |

**개인화 모델**: Ridge Regression vs Random Forest (교차검증으로 자동 선택)
- 저장 위치: `models/calorie_model_{user_id}.pkl`
- 활성화: 30개 이상 식사 기록 누적 시
- 재학습: 매주 일요일 03:00 (APScheduler)

**데이터 누적 현황**: `meals.gpt_comment` + `meals.calories` → 학습 레이블 역할

## gpt_ml_bridge.py — ML→GPT 브릿지

**역할**: ML 결과를 GPT `extra_context` 파라미터로 주입

**주요 함수**:
- `get_corrected_calories(user_id, food_name, meal_type, gpt_calories)` — 실제 호출 중
- `generate_comment_with_pattern(...)` — **현재 미사용 (데드코드)**. 파라미터 불일치로 추후 수정 필요

### 메일봇 (bot_mail.py)

| 단계 | 기능 | 방법 | 조건 |
|------|------|------|------|
| 현재 | 이메일 요약 | GPT (summarize_email) | — |
| 임시 최적화 | 200자 이하 원문 표시 | 규칙 기반 (v3.1 적용) | — |
| ML 1단계 | 스팸 분류 고도화 | Naive Bayes / SVM | `email_log.is_spam` 누적 후 |
| ML 2단계 | 이메일 요약 모델 | Extractive → Abstractive | 수천 건 누적 후 |

**스팸 분류 고도화 상세**
- 현재: 키워드 하드코딩 (`[광고]`, `[홍보]` 등) — 개인 맞춤 불가
- ML 전환: 유저가 직접 스팸 표시한 메일 데이터로 개인화 학습
- 개인마다 다른 "스팸 기준"을 반영 (e.g. 한 유저는 카페 뉴스레터 OK, 다른 유저는 스팸)
- 데이터 누적: `email_log.is_spam` (유저 피드백 레이블)

### 식사봇 (bot_meal.py, 예정)

| 단계 | 기능 | 방법 | 조건 |
|------|------|------|------|
| 현재 | 칼로리 보정 | Ridge/RF (ml.py) | 30건+ |
| ML 2단계 | 음식 추천 개인화 | Collaborative / Content-based Filtering | 식사 이력 충분 시 |
| ML 3단계 | 한식 이미지 인식 | CNN MobileNetV3 (AI Hub) | 사진 데이터 충분 시 |

**음식 추천 개인화 상세**
- n8n이 추천한 결과 중 유저가 실제로 선택/기록한 음식 패턴을 학습
- "이 유저는 점심에 한식 선호, 매운 음식 선호, 커피와 함께 먹는 경향" 등 파악
- 감정 상태(일기봇 데이터)와 결합하면 기분에 맞는 추천으로 고도화 가능
- 데이터 누적: `meals.food_name` + 선택 이력

### 일기봇 (bot_diary.py, 예정)

| 단계 | 기능 | 방법 | 조건 |
|------|------|------|------|
| 초기 | 감정 분석 | GPT (레이블 누적) | — |
| ML 전환 | 감정 분류 모델 | KoBERT / 감정 분류기 | 수백 건 누적 후 |
| 고도화 | 식사 × 감정 상관관계 분석 | 교차 분석 + 패턴 탐지 | diary + meals 동시 누적 |

**감정 분석 상세**
- 일기 텍스트 → 긍정/부정/중립 + 세부 감정 키워드 분류
- GPT가 먼저 레이블 생성 → DB 누적 → KoBERT fine-tuning으로 GPT 대체

**식사 × 감정 상관관계 활용처**

이 기능은 단독 기능이 아니라 다른 기능들을 똑똑하게 만드는 데이터 레이어:

```
① 과식 패턴 사전 경고
   학습된 패턴: "스트레스 받은 날 → 저녁 칼로리 +40%"
   오늘 일기에서 부정 감정 감지
   → 먹구름봇: "오늘 좀 힘든 날이네. 너 스트레스 받으면
      저녁에 많이 먹는 경향이 있거든, 오늘 저녁 조심해봐 👀"

② 주간 리포트 인사이트 강화
   기존: "이번 주 평균 칼로리 1,850kcal"
   강화: "화요일에 칼로리가 제일 높았는데,
          그날 일기 보니까 많이 힘들었던 날이었네.
          기분이 안 좋을 때 더 먹는 패턴이 보여 🍔"

③ 음식 추천에 감정 반영
   n8n 추천 payload에 emotion 필드 추가:
   스트레스 상태 → 위로 음식 / 자극적이지 않은 것 추천
   기분 좋은 날  → 건강식 추천 (받아들일 가능성 높음)

④ 먹구름봇 대사 맥락화
   기존: "오늘 치킨 먹었구나~ 맛있었겠다!"
   강화: "오늘 힘든 하루였는데 치킨으로 위로했구나.
          가끔은 괜찮아, 내일 다시 시작하면 돼 🍗"
```

데이터 누적: `diary_log.emotion` × `meals` 조인

### 먹구름봇 (bot.py) — GPT + ML 하이브리드 전략

| 기능 | 현재 | 목표 |
|------|------|------|

> 복합 의도 파악이나 엔티티 추출은 GPT 유지.

## 의도 분류기 (utils/intent_classifier.py — v4.0 예정)

**역할**: 유저 발화에서 어느 전문봇을 트리거할지 분류

**학습 데이터**: `intent_log` 테이블 (GPT가 분류한 결과 누적)

```python
# utils/intent_classifier.py
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
import pickle

class IntentClassifier:
    """
    유저별 개인화 의도 분류기.
    활성화 조건: 유저당 intent_log 50건+
    재학습: 매주 일요일 03:30 (ML 재학습과 동일 주기)
    저장: models/intent_{user_id}.pkl
    """
    def predict(self, user_id: str, text: str) -> str | None:
        # None 반환 시 GPT fallback
        ...
```

**GPT → ML 전환 흐름**:

```
1. 유저가 전용 채널에 메시지 입력
2. ML 분류기 predict() 시도
   ├── 모델 있음 → ML 의도 분류 (빠름, 비용 0)
   └── 모델 없음(50건 미만) → GPT 의도 분류
3. GPT 분류 시: intent_log 저장 (학습 데이터 축적)
4. 50건+ 누적 → 매주 재학습 → ML 전환 완료
5. ML 전환 후: GPT는 엔티티 추출만 사용
```

**ML 로드맵**:

| 단계 | 기능 | 방법 | 조건 |
|------|------|------|------|
| 현재 | 의도 분류 | GPT (레이블 누적) | — |
| ML 전환 | 의도 분류 | TF-IDF + LogisticRegression | 유저당 50건+ |
| 고도화 | 복합 의도 처리 | Multi-label 분류 | 수백 건+ |
